# Weight-List V1 Design Spec

## Overview

Weight-List is a tool that prioritizes inbound leads using AI-powered weighted scoring. Users describe what they're evaluating, the AI generates scoring criteria, and then scores uploaded CSV data against those criteria — grouping results into tiered buckets (Strong / Possible / Weak).

**Use cases:** Sales lead prioritization, recruitment/job application sifting, supplier evaluation (procurement).

## Prerequisites

Install AI SDK packages before development:

- `ai` (Vercel AI SDK)
- `@ai-sdk/anthropic` (Anthropic provider)
- `zod` (schema validation)
- `papaparse` + `@types/papaparse` (CSV parsing)

Environment variable: `ANTHROPIC_API_KEY` in `.env.local`. Create a `.env.example` with the key name for documentation.

## User Flow

Four steps on a single page with step progression. Each step shows a loading/pending state while AI calls are in flight (spinner + disabled controls).

### Step 1: Describe

Single text area where the user writes in plain language what they're scoring for and what they prioritize. Example: "I'm hiring a senior frontend developer. I care most about React experience, portfolio quality, and salary expectations."

User hits "Generate Scoring Criteria" and the AI returns a structured rubric.

### Step 2: Configure

Editable rubric UI generated by the AI. Each criterion has:

- **Label** — editable text input (user can rename)
- **Weight** — editable percentage (user can adjust)
- User can **add** or **remove** criteria
- Weights must sum to 100%. If they don't, the Confirm button is disabled with an inline message showing the current total and the delta.
- The AI is instructed via prompt to always return weights summing to 100%. If despite this the initial rubric doesn't sum correctly, auto-normalize the weights on arrival before displaying.
- User confirms when satisfied

### Step 3: Upload & Map

- Drag-and-drop or click-to-upload CSV file
- CSV is parsed **client-side** via `papaparse` (FileReader API). Parsed column headers and row data are sent to the server action — no file upload to the server.
- AI analyzes CSV columns and proposes a mapping: CSV column -> scoring criterion
- Displayed as a review table with columns: Criterion | Mapped Column | Confidence (High / Medium / Low)
- The AI also proposes which column to use as the **lead identifier** (the human-readable name/label for each row). User can override via dropdown.
- User can override any mapping via dropdown select
- User confirms when satisfied

### Step 4: Results

- AI scores each lead against confirmed criteria + mappings
- All scoring happens inside a single server action call. Within that call, rows are batched into groups of 10 and each batch is scored via `Promise.all`. This is server-side parallelism — the client makes one action call and waits.
- V1 caps input at 200 rows — displays a warning if the CSV exceeds this.
- All scores use a 0–100 scale. Each criterion gets an individual score (0–100). Weighted contribution = individual score x weight / 100. Overall score = sum of all weighted contributions.
- Results displayed as three tiered buckets: **Strong** (70+), **Possible** (40-69), **Weak** (below 40)
- Each tier is a collapsible group showing count
- Each lead within a tier shows: lead identifier (from mapped identifier column), overall score
- Expandable per-criterion breakdown: criterion name, weight, individual score, weighted contribution

## Error Handling

V1 uses simple inline error messages at each step. If an AI call fails, show "Something went wrong. Please try again." with a retry button. If CSV parsing fails, show "Could not parse this file. Please check the format and try again." Detailed errors go to `console.error()`.

## Technical Architecture

### Stack

- Next.js 16 (App Router)
- Tailwind v4 + shadcn + Radix UI
- Vercel AI SDK with Anthropic provider (`@ai-sdk/anthropic`)
- Zod for schemas and type inference
- No auth for V1 (Clerk later)
- No database for V1 (Neon serverless Postgres later)
- All state is ephemeral (React client state)

### AI Integration

Single AI service file (`lib/ai/ai.service.ts`) exporting async functions:

- `generateRubric(description: string)` — uses `generateObject` with Zod schema. Takes plain-language input, returns structured criteria with weights summing to 100%. Prompt instructs the model to always produce valid weights.
- `mapColumns(criteria, csvColumns)` — uses `generateObject` with Zod schema. Takes criteria and CSV column headers, returns proposed mappings with confidence levels (High/Medium/Low) and a proposed identifier column.
- `scoreLeads(criteria, mappings, rows)` — uses `generateText` with web search tool + structured prompt. Web search is intentionally included here (unlike `generateRubric` and `mapColumns`) because scoring benefits from real-time context: looking up a company's website, verifying a candidate's background, or checking a supplier's reputation. The tool uses `anthropic.tools.webSearch_20250305({ maxUses: 2 })` — capped at 2 searches per batch to limit latency and cost. The prompt instructs the model to return a JSON block containing an array of scored leads. Each lead has an overall score (0–100) and per-criterion scores (0–100). The response is parsed by extracting the JSON block from the text output and validating it against a Zod schema in `ai.schemas.ts`. Parsing logic lives in `ai.service.ts`. Rows are batched (10 per batch) and batches run via `Promise.all` within the single action call.

**Schema boundary rule:** `lib/ai/ai.schemas.ts` contains the Zod schemas that define the raw shape returned by the AI model (the `schema` param passed to `generateObject`, or the expected JSON shape from `generateText`). `models/` contains the Zod schemas that define the shapes used by the rest of the app (actions, components, state). When the AI output shape and the domain shape are identical, the domain schema re-exports from the AI schema to avoid duplication.

### Server Actions

Follow `entity.action.ts` naming convention in the `actions/` folder at the project root (not inside `app/`). All action files use the file-level `'use server'` directive at the top of the file (not inline per function).

- `actions/rubric.action.ts` — exports `generateRubricAction`. Single function; rubric editing (rename, reweight, add/remove) is purely client-side state — no server action needed for updates.
- `actions/mapping.action.ts` — exports `mapColumnsAction`. Single function.
- `actions/scoring.action.ts` — exports `scoreLeadsAction`. Single function. All batching and parallelism happens inside this one call.

Each action uses Zod validation and returns user-friendly errors. No auth checks for V1.

### CSV Parsing

`lib/csv/csv.parser.ts` — thin wrapper around `papaparse`. Called client-side. Extracts column headers and row data as plain objects. Handles varying delimiters and header formats.

### Project Structure

```
actions/                          — server actions at project root
  rubric.action.ts
  mapping.action.ts
  scoring.action.ts
models/                           — domain Zod schemas + inferred types at project root
  rubric.schema.ts
  mapping.schema.ts
  scoring.schema.ts
lib/
  ai/
    ai.service.ts                 — AI functions (generateRubric, mapColumns, scoreLeads)
    ai.schemas.ts                 — Zod schemas for raw AI model output
  csv/
    csv.parser.ts                 — CSV parsing wrapper
app/
  page.tsx                        — single page, step orchestration
  _components/
    describe.tsx                  — step 1
    configure.tsx                 — step 2
    upload-map.tsx                — step 3
    results.tsx                   — step 4
components/
  ui/                             — shadcn shared components
```

### Key Conventions

- Types inferred from Zod schemas via `z.infer` — no separate type/interface definitions
- Action files: file-level `'use server'` directive, `entity.action.ts` naming
- Step components colocated with the page under `app/_components/`
- Shared UI components (shadcn) in `components/ui/`
- AI schemas (raw model output shapes) in `lib/ai/ai.schemas.ts`
- Domain schemas (app data shapes) in `models/` — re-export from AI schemas when identical
- Environment variable: `ANTHROPIC_API_KEY` in `.env.local`

## Design Direction

Bold, clean, accessible. Strong visual design using shadcn + Tailwind. No specific mockups — design decisions made during implementation with emphasis on clarity and impact.
